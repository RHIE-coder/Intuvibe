Plan은 Spec + Architecture + Test Strategy를 통합한다. Architecture는 small 규모에서 생략 가능. Code는 TDD — Test 작성 후 구현.

세 가지 "설계" 단계 — Spec / Architect / Plan

Spec · Architect · Plan 은 모두 설계지만 다루는 질문이 다르다. 섞으면 대부분의 설계 실패가 여기서 시작된다.

Spec
WHAT — 무엇을 만드는가
관점: 사용자·제품
산출물: specs/{domain}/{feature}.spec.yaml — AC 리스트
다루지 않는 것: 기술 스택·DB 스키마·API 경로
게이트: 모든 AC가 testable
에이전트: requirements-analyst · test-strategist · devils-advocate

예: AC-001: 이메일+비밀번호 일치 시 JWT 발급
Architect
HOW (system) — 시스템으로 어떻게
관점: 아키텍처·장기 유지보수
산출물: decisions/{nnn}-{title}.md — ADR
다루지 않는 것: 개별 함수·step 분해
게이트: medium+ 필수 · SOLID/OWASP/유저 선호 반영
에이전트: architect · reviewer-security

예: 003-auth-strategy.md — JWT(A15m+R7d) 채택, Redis 세션 기각
Plan
HOW (task) — 태스크로 어떻게
관점: 구현자·단기 실행
산출물: plans/{domain}/{feature}.plan.md — step 리스트
다루지 않는 것: "왜 필요한가"·"이 패턴이 옳은가"
게이트: 모든 AC ↔ step 매핑 (coverage.json)
에이전트: planner · implementer(consumer)

예: Step 2: bcrypt 검증 + JWT 서명 (AC-001)

자주 하는 혼동: Plan에 아키텍처 판단 섞기(ADR 없어 회고·재사용 불가) · Spec에 API 경로 쓰기(Architect 여지 상실) · medium+에서 Architect 건너뛰기(Plan 간 일관성 붕괴) · 세 개를 한 문서에 몰기(리뷰 단위 비대화).

한 줄 요약: Spec은 유저가 만족하는가, Architect는 시스템으로 지속 가능한가, Plan은 실제로 만들어지는가 를 각각 증명한다.

Glossary — 반복 용어

AC
Acceptance Criteria
Spec이 "완성됐다"고 선언할 수 있는 검증 가능한 조건 목록. Spec↔Test↔Plan↔Code를 연결하는 핵심 ID.

4조건: Testable / Atomic / Binary / User-visible
예: AC-002: 비밀번호 불일치 시 401 + error='invalid_credentials'
추적: .harness/state/coverage.json — AC ↔ plan step ↔ test 3자 매핑
게이트: spec(testable 필드) · plan G2(매핑) · implement G3(test 존재) · qa G4(모든 test PASS)
Right-Size
규모에 맞는 ceremony
작업 규모(small / medium / large)에 따라 프로세스 무게를 조절. Iron Laws는 규모 무관 영구 강제, 그 외 선택 절차만 차등.

판정(3축 AND · 결정론적): small=AC≤3 ∧ 파일≤4 ∧ LOC≤100 · medium=AC≤7 ∧ 파일≤10 ∧ LOC≤400 · large=그 외
근거: Miller 1956 (3±1 / 7±2), SmartBear 2012 (≤4 files, 400 LOC/h), Rigby & Bird 2013 (PR median ~11 files), Google eng-practices (100/400 LOC).
차등: Architect 필수(medium+) · Review 깊이(light/standard/deep) · QA 테스트 매트릭스(9종 중 Right-Size별 조합)
불변: Safety · Spec→Test→Plan 순서 · 구현자≠검증자 · QA→Deploy
Mode와 직교: Mode=게이트 on/off, Right-Size=게이트의 강도

→ 전체 임계값 표 · 판정식 · 구현 흐름: Workflow 탭 § Right-Size Ceremony (전 9종 테스트 매트릭스 포함)
QA Stack
계층 의존 순차 검증 (Bottom-Up)
UI·API·DB·Infra는 의존 스택. 하위 PASS + 상위 FAIL ⇒ 원인은 반드시 그 상위 계층. 하네스가 실패 귀인을 자동화.

기본 스택: Infra(L0) → DB(L1) → API(L2) → UI(L3)
모드 (testing.qa_stack.mode): parallel (기본, 빠름) · sequential_bottom_up (엄격, 자동 귀인) · custom
Attribution: scripts/qa/attribution.mjs → .harness/state/qa-attribution.json — verdict: pure_{layer}_issue | multi_layer | all_green
귀인 신뢰 전제 (mock_guard): 상위가 하위 mock → 전제 무효. strict_lower_real(기본)=G4 BLOCK · warn · off
Right-Size 권고: small=parallel · medium=상황 적응 · large=sequential_bottom_up
Audit: qa_layer_halted · qa_attribution_report · qa_attribution_warning
→ 스택 다이어그램 · 흐름 · 모드 비교: Workflow 탭 § QA Stack
Mode Auto-Detect
세션 시작 시 모드 자동 판별 (opt-in)
config.mode: auto 일 때만 동작. SessionStart 시점 determine-mode.mjs가 브랜치·스펙·첫 프롬프트 신호로 1회만 판별.

왜 opt-in: mode는 게이트 on/off를 좌우 — 오판 시 게이트 조용히 풀림. dangerously-skip-permissions와 유사한 명시적 opt-in.
안전 3규칙: narrow_only (standard 자동 내림 금지) · promotion_requires_user (승격은 /harness:sync만) · session_start_only (중간 전환 금지)
판정: RULE-MODE-EXPLORE-001 → RULE-MODE-PROTOTYPE-001 → RULE-MODE-STANDARD-DEFAULT (fallthrough)
기록: 모든 전환은 audit.jsonl에 mode_auto_detected 이벤트로
Audit Log
자동 판단의 역추적 장치
하네스가 자동으로 내린 모든 판단을 기록. 저장: .harness/state/audit.jsonl (append-only JSONL).

목적: (1) Audit — 왜 이 모드/크기로 확정됐는가, 왜 게이트 skip됐는가 (2) Debug — 자동 판단 오작동 시 입력 신호·적용 규칙 역추적

Design Philosophy

하네스의 모든 설계 결정은 이 원칙들로부터 도출된다. 3개 프레임워크(Superpowers, bkit, Compound Engineering) 교차 분석에서 도출.

10 Design Principles

P1
Safety First
안전이 먼저, 기능은 그 다음. Safety Layer는 Config으로도 끌 수 없다. 파괴적 작업은 구조적으로 차단.

SP(Iron Laws) + bkit(Guardrail-as-Code) — 2/3 수렴
P2
Distrust by Structure
구현과 검증의 구조적 분리. 같은 에이전트가 두 역할을 맡는 것이 물리적으로 불가능해야 한다.

SP + bkit + CE — 3/3 수렴 (가장 강한 패턴)
P3
Spec is Source of Truth
코드는 blackbox, Spec이 기준. Spec이 없으면 Test도, Plan도, Code도 존재 이유가 없다.

사용자 요구 + CE(R-ID tracing)
P4
Test Before Code
Spec에서 Test를 먼저 도출. FAIL에서 시작하여 구현으로 PASS. 테스트 유효성도 검증.

SP(TDD for Everything) — RED-GREEN-REFACTOR
P5
Controller Curates
서브에이전트에게 탐색을 시키지 않는다. Controller가 컨텍스트를 구성하여 전달.

SP + bkit + CE — 3/3 수렴
P6
Earn, Don't Assume
자동화는 실적으로 획득. 처음에는 모든 것을 강제하고 확인. 기본값은 항상 "수동 확인".

bkit(Progressive Automation L0-L4)
P7
Token Budget is Real
Script-First로 65% 절감. 합리화 방어는 Lazy Loading. 프롬프트 비대화 방지.

CE(Script-First) + SP(7.8x 초과 교훈)
P8
Right-Size Everything
Iron Laws는 강제 + 절차 수준은 규모에 따라 조절. 작은 작업에 과도한 프로세스 = 마찰.

CE(Right-Size) vs SP(7-Stage 강제) — 둘 다 맞음
P9
Escalate, Don't Struggle
확신 없이 진행하는 것이 가장 위험. 정직한 에스컬레이션 > 불확실한 결과물.

SP + bkit + CE — 3/3 수렴
P10
Start Simple, Grow Proven
단순한 구현에서 시작. 복잡도는 필요가 증명된 후에만 추가. 파일 기반 → 필요 시 FSM.

Config, 사용자 요청, 어떤 수단으로도 비활성화할 수 없는 규칙.

IL-7 빈 문서 금지

6-레이어 아키텍처와 강제 메커니즘

Feature = Aggregate — 일관성 경계 사고법

DDD의 Aggregate 어휘를 차용 (코드 프레임워크 도입 X). 하네스의 상태·규칙·불변식의 최소 단위는 feature (예: auth/login). 모든 일관성 규칙은 feature 경계 안에서만 즉시 성립하고, 경계 간에는 eventual consistency. 근거: docs/insights/patterns/01-ddd-aggregate-actor.md, 05-harness-applicability.md §4.1.

부록 A — 테스팅 방법 정리 (9종 비교)

"무엇을 묻는 테스트인가" 기준의 차이. 상세: 03-workflow.md 부록 A.

Unit
"이 함수가 맞게 동작하는가?"
범위: 단일 함수/클래스 (mock 허용)
속도: 초 이내 · 코드 변경마다 실행
오해: "1 test = 1 assert"가 아니라 1 단위의 1 행동
예: AuthService.login() 반환 검증 (DB mock)
Sanity
"수정한 부분이 말이 되는가?"
범위: 변경 부위 국소 — 좁고 얕음
트리거: 버그 수정·핫픽스 직후
Smoke와 차이: Smoke는 넓고 얕음(핵심 경로), Sanity는 좁고 얕음(수정 부위)
예: lockout 수정 → 해당 함수만 호출해 0 아닌 값 반환 확인
Smoke
"불 켜니 연기 나는가?"
범위: 핵심 경로 1~2개 E2E 슬라이스 — 넓고 얕음
트리거: 빌드/배포 직후
통과 조건: business-critical flow가 end-to-end로 돌아감
예: 로그인 → 메인 → 로그아웃 스크립트 1회 통과
Integration
"모듈들이 함께 동작하는가?"
범위: 2+ 모듈 + 실제 의존성(DB, Redis, 외부 API)
Unit과 차이: mock 대신 컨테이너/실제 인스턴스
검증 대상: 모듈 간 계약·트랜잭션·타이밍
예: AuthService + Postgres + Redis — 5회 실패 후 Redis lockout 카운터 반영
Regression
"예전에 되던 게 아직도 되는가?"
범위: 전체 기존 test suite 재실행 (비파괴 검증)
원칙: 버그 수정 시 재현 test를 먼저 추가 → 영구 회귀 방어
하네스 강제: 기존 test 삭제/수정 시 경고 + 사유 기록
예: PR마다 CI에서 전체 suite 실행 · PostToolUse Hook side-effect 검출
E2E
"유저 시나리오가 끝까지 돌아가는가?"
범위: UI → 백엔드 → DB → 외부 시스템 전 구간
Smoke와 차이: Smoke는 핵심 1개, E2E는 다양한 시나리오 + edge case
도구: Playwright / Cypress / Selenium
예: 가입 → 이메일 인증 → 로그인 → 결제 → 환불 → 로그아웃
Load
"기대 부하에서 SLA를 지키는가?"
범위: 시스템 전체 — 평소 수준 부하
통과 조건: p95/p99 latency · error rate · 리소스 사용량 임계 이내
Stress와 차이: Load = "평소 OK", Stress = "한계 초과 시"
예: k6로 1000 RPS · 60s · p95 < 200ms · error < 0.1%
Stress
"한계 넘기면 어떻게 망가지는가?"
범위: 시스템 전체 — 초과 부하
목적: "안 망가짐"이 아닌 "망가질 때 graceful한가"
통과 조건: 명확한 에러 응답 · 데이터 손상 無 · circuit breaker · 자동 회복
예: 10000 RPS → OOM 없이 429 반환 → 부하↓ 시 30초 내 정상화
Recovery
"장애 후 복구되는가?"

Confidence Rubric (Evidence-based)

"LLM 자가 신뢰도"는 신뢰할 수 없다. 하네스는 5개 카테고리 체크리스트로 confidence를 계산한다. 에이전트는 카테고리별 점수를 emit하고 Controller가 가중합을 재계산한다 (자가 부풀리기 방지).

카테고리가중치0.00.51.0
E Evidence (직접 관측)0.30추론만패턴 유추 + 일부 확인파일/코드/spec 직접 인용
C Coverage (AC 범위)0.20AC 일부 미처리주요 AC + 일부 edge모든 AC + 주요 edge
V Verification (검증 실행)0.30미실행정적 검증만 (type/lint)test 실제 실행 PASS
X eXternal deps0.10미확인문서 참조로 가정실제 호출/버전 확인
A Assumption log0.10가정 미기록일부 기록모든 가정 + 검증 방법

confidence = 0.30·E + 0.20·C + 0.30·V + 0.10·X + 0.10·A  →  ≥0.80 DONE · 0.60~0.80 CONCERNS · <0.60 NEEDS_CONTEXT

예: 구현 완료 but test 미실행 → E=1.0, C=1.0, V=0.5, X=0.5, A=0.5 → 0.75 CONCERNS

/harness:qa 플래그 전체 스펙

가장 복잡한 스킬. Right-Size 매트릭스 × CLI 플래그 × QA Stack mode가 직교.

플래그용도예시
(없음)config.testing.scope.mode(기본 auto → Right-Size 매트릭스)/harness:qa auth/login
--all정의된 모든 유형 실행/harness:qa --all
--only <types>지정 유형만 (매트릭스 무시)/harness:qa --only unit,smoke
--exclude <types>기본 집합에서 제외/harness:qa --exclude load,stress
--skip <types>--exclude 별칭 (하위호환)/harness:qa --skip load
--force-size <s>이번 호출 1회만 right-size (workflow.json 불변)/harness:qa --force-size large
--reason "..."필수 skip 시 사유 (audit + qa-log 기록)/harness:qa --skip load --reason "infra maint"
--stack-mode <m>QA Stack 실행 모드 1회 override/harness:qa --stack-mode sequential_bottom_up

우선순위: --only > --all > --exclude/--skip. --force-size는 매트릭스 재계산, --stack-mode는 순서만 변경.
Exit codes: 0=G4 PASS · 1=FAIL · 2=block_deploy 정책상 차단.
정책: skip_required_policy: block_deploy | warn | allow — 필수 유형 skip 시 G4 거동 제어.

플래그 네이밍 일관성

--force-*
파라미터 상향 override
Iron Law 우회 아님. upward_only.
등장: architect --force-size, qa --force-size
Audit: force_size_applied
--from-*
상류 산출물 참조
spec --from-brainstorm, plan --from-spec, sync --from <commit>
--reason "..."
정책 우회 시 사유 기록 강제
생략 시 block_deploy 정책상 BLOCK.
audit.jsonl + .harness/state/qa-log.jsonl 양쪽 기록.
--dry-run
실제 변경 없이 제안/점검
등장: deploy --dry-run, sync --dry-run
--bypass-* (Escape Hatch)
Safety/Gate 의도적 우회
Claude Code --dangerously-skip-permissions 대응. 이름이 아닌 구조로 위험 강제.
계열: --bypass-gates <g1,...> · --bypass-review · --bypass-qa · --bypass-deploy-gate
3중 안전장치: ① --reason 필수 (생략 시 exit 2) ② gate_bypassed audit ③ 세션 sticky warning
불가변: 파괴적 bash · force-push · .harness/state/ 편집 · prototype deploy
