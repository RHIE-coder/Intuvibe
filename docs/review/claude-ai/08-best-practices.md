# Best Practices — 효과적인 Claude Code 사용 패턴

---

Claude Code는 채팅봇이 아니라 **에이전틱 코딩 환경**이다. 파일을 읽고, 명령을 실행하고, 변경을 만들고, 문제를 자율적으로 해결한다.
이 자율성을 최대로 활용하기 위한 핵심 패턴과 안티패턴을 정리한다.

**모든 베스트 프랙티스의 근본 제약: 컨텍스트 윈도우는 빠르게 채워지고, 채워질수록 성능이 하락한다.**

---

## 1. 검증 수단을 제공하라 — 가장 레버리지 높은 행동

Claude가 스스로 작업을 검증할 수 있으면 성능이 **극적으로** 향상된다.

| 전략 | 나쁜 예 | 좋은 예 |
|------|--------|--------|
| **검증 기준 제공** | "이메일 검증 함수 만들어" | "validateEmail 작성. 테스트: user@example.com → true, invalid → false. 테스트 실행해" |
| **UI 시각적 검증** | "대시보드 개선해" | "[스크린샷] 이 디자인 구현해. 결과 스크린샷 찍고 원본과 비교. 차이점 나열 후 수정" |
| **근본 원인 해결** | "빌드 실패함" | "빌드가 이 에러로 실패: [에러]. 고치고 빌드 성공 확인. 에러 억제가 아닌 근본 원인 해결" |

검증 수단 = 테스트 스위트, 린터, Bash 명령, 스크린샷 비교, Chrome 확장. **검증이 견고할수록 결과가 좋다.**

---

## 2. 탐색 → 계획 → 구현 → 커밋

```
Plan Mode                             Normal Mode
────────────────────────────────      ────────────────────────────────
1. Explore: 파일 읽기, 질문          3. Implement: 코드 작성, 테스트
2. Plan: 구현 계획 수립              4. Commit: 커밋 + PR
   (Ctrl+G로 에디터에서 편집)
```

**Plan Mode를 건너뛸 때:** 스코프가 명확하고 변경이 작을 때 (오타 수정, 로그 추가, 변수 이름 변경). diff를 한 문장으로 설명할 수 있으면 계획 불필요.

**Plan Mode가 필요할 때:** 접근 방법이 불확실, 여러 파일 수정, 익숙하지 않은 코드.

## 3. 프롬프트에 구체적 컨텍스트를 제공하라

| 전략 | 나쁜 예 | 좋은 예 |
|------|--------|--------|
| **스코프 지정** | "foo.py에 테스트 추가" | "foo.py에 로그아웃 상태 엣지 케이스 테스트 작성. mock 피해" |
| **소스 지목** | "ExecutionFactory API가 왜 이상해?" | "ExecutionFactory의 git 히스토리 보고 API가 어떻게 형성됐는지 요약" |
| **패턴 참조** | "캘린더 위젯 추가" | "홈페이지 위젯 구현 패턴 확인. HotDogWidget.php가 좋은 예. 같은 패턴으로 캘린더 위젯 구현" |
| **증상 설명** | "로그인 버그 고쳐" | "세션 타임아웃 후 로그인 실패. src/auth/의 토큰 리프레시 확인. 실패 테스트 먼저 작성 후 수정" |

**리치 콘텐츠 제공 방법:**
- `@`로 파일 참조
- 이미지 붙여넣기
- URL 제공 (자주 쓰는 도메인은 `/permissions`로 허용)
- `cat error.log | claude`로 파이프
- Claude에게 직접 가져오게 하기 (Bash, MCP, 파일 읽기)

## 4. 환경 설정

### CLAUDE.md 작성 원칙

`/init`으로 시작 → 점진적 다듬기.

| 포함 | 제외 |
|------|------|
| Claude가 추측 못하는 Bash 명령 | 코드 읽으면 알 수 있는 것 |
| 기본값과 다른 코드 스타일 | 표준 언어 규칙 |
| 테스트 지침, 선호 테스트 러너 | 상세 API 문서 (링크로 대체) |
| 레포 에티켓 (브랜치명, PR 규칙) | 자주 변하는 정보 |
| 프로젝트 아키텍처 결정 | 긴 설명/튜토리얼 |
| 개발 환경 특이사항 (필수 env vars) | 파일별 코드베이스 설명 |
| 비자명한 gotcha | "깨끗한 코드 작성" 같은 자명한 것 |

**핵심 테스트:** 각 줄에 "이거 빼면 Claude가 실수하나?" → 아니면 삭제.

강조 필요 시 "IMPORTANT" 또는 "YOU MUST" 사용으로 준수율 향상 가능.

### 권한 설정

3가지 중단 감소 수단:
- **Auto mode** — classifier가 위험한 것만 차단
- **Permission allowlists** — 안전한 명령 사전 허용 (`npm run lint`, `git commit`)
- **Sandboxing** — OS 레벨 파일시스템/네트워크 격리

### CLI 도구 활용

`gh`, `aws`, `gcloud`, `sentry-cli` 등 설치 → Claude가 가장 컨텍스트 효율적인 방법으로 외부 서비스 접근.
모르는 CLI도 `--help`로 학습 가능.

### MCP, Hooks, Skills, Subagents, Plugins

- **MCP** — 외부 서비스 연결 (`claude mcp add`)
- **Hooks** — 예외 없이 매번 실행되어야 하는 작업 (CLAUDE.md와 달리 결정론적 보장)
- **Skills** — 온디맨드 도메인 지식/워크플로우 (`disable-model-invocation: true`로 사이드이펙트 스킬 보호)
- **Subagents** — 격리된 전문 워커 (도구 접근 제한, 별도 컨텍스트)
- **Plugins** — `/plugin`으로 커뮤니티/Anthropic 번들 설치. code intelligence 플러그인으로 심볼 네비게이션

## 5. 효과적인 커뮤니케이션

### 코드베이스 질문

시니어 엔지니어에게 하듯 질문. 특별한 프롬프트 불필요:
- "로깅은 어떻게 되어 있어?"
- "새 API 엔드포인트는 어떻게 만들어?"
- "foo.rs 134줄의 `async move { ... }`는 뭐야?"

### Claude 인터뷰 패턴

큰 기능을 만들 때, Claude에게 먼저 인터뷰를 받아라:

```
[간략 설명]을 만들고 싶어. AskUserQuestion 도구로 자세히 인터뷰해.
기술 구현, UI/UX, 엣지 케이스, 우려사항, 트레이드오프에 대해 물어봐.
뻔한 질문 말고, 내가 놓쳤을 어려운 부분을 파고들어.
모든 걸 다룰 때까지 인터뷰한 후 SPEC.md에 완전한 스펙 작성.
```

스펙 완성 → **새 세션에서 실행** (깨끗한 컨텍스트로 구현 집중).

## 6. 세션 관리

### 빠른 교정

- `Esc` — 중단 (컨텍스트 유지)
- `Esc + Esc` 또는 `/rewind` — 이전 체크포인트로 되돌리기/요약
- `"Undo that"` — Claude에게 되돌리기 지시
- `/clear` — 무관한 작업 사이에 컨텍스트 리셋

**같은 이슈를 2번 이상 교정했으면** → `/clear` + 학습 내용을 반영한 더 나은 프롬프트로 재시작. 교정 누적보다 클린 세션이 거의 항상 더 낫다.

### 컨텍스트 공격적 관리

- `/clear` — 작업 사이에 자주 사용
- `/compact <지시>` — 포커스 지정 압축 ("API 변경에 집중")
- `Esc + Esc` → "Summarize from here" — 특정 지점부터 부분 압축
- CLAUDE.md에 compact 지침: "compact할 때 수정된 파일 목록과 테스트 명령은 반드시 보존"
- `/btw` — 컨텍스트에 안 남는 사이드 질문 (dismissible overlay)

### 서브에이전트로 리서치 위임

```
서브에이전트를 사용해서 인증 시스템의 토큰 리프레시 처리와
기존 OAuth 유틸리티 재사용 가능 여부를 조사해.
```

서브에이전트가 탐색 → 요약만 반환 → 메인 컨텍스트 깨끗하게 유지.
구현 후 검증에도 사용: `"서브에이전트로 이 코드의 엣지 케이스 리뷰해"`

### 체크포인트 활용

모든 Claude 액션은 체크포인트 생성. `Esc + Esc` 또는 `/rewind`로 대화/코드/둘 다 복원 가능.
**위험한 시도를 두려워하지 마라** — 안 되면 되돌리면 된다. 세션 종료 후에도 체크포인트 유지.

## 7. 자동화와 스케일링

### 비대화형 모드

```bash
claude -p "prompt"                              # 원샷 쿼리
claude -p "prompt" --output-format json          # 구조화 출력
claude -p "prompt" --output-format stream-json   # 실시간 스트리밍
```

CI, pre-commit hooks, 자동화 워크플로우에 통합.

### 병렬 세션

| 방법 | 특징 |
|------|------|
| Desktop app | 시각적 관리, 각 세션 격리 worktree |
| Claude Code on the web | Anthropic 클라우드 VM |
| Agent teams | 공유 작업 목록 + 메시징으로 자동 조율 |

**Writer/Reviewer 패턴:**
- 세션 A (Writer): 구현
- 세션 B (Reviewer): 독립 컨텍스트에서 리뷰 (자기가 쓴 코드 편향 없음)
- 세션 A: 리뷰 피드백 반영

### 파일 단위 팬아웃

```bash
for file in $(cat files.txt); do
  claude -p "Migrate $file from React to Vue. Return OK or FAIL." \
    --allowedTools "Edit,Bash(git commit *)"
done
```

2-3개 파일로 테스트 → 프롬프트 다듬기 → 전체 실행. `--allowedTools`로 무인 실행 시 권한 제한.

### Auto 모드 자동화

```bash
claude --permission-mode auto -p "fix all lint errors"
```

비대화형(`-p`)에서 classifier 반복 차단 시 세션 중단.

## 8. 안티패턴 — 피해야 할 실수

| 안티패턴 | 문제 | 해결 |
|---------|------|------|
| **주방 싱크 세션** | 하나의 세션에서 무관한 작업을 섞음 → 컨텍스트 오염 | `/clear`로 작업 사이 리셋 |
| **반복 교정** | 같은 문제를 2번 이상 교정 → 실패 접근이 컨텍스트 오염 | `/clear` + 학습 반영한 새 프롬프트 |
| **과잉 CLAUDE.md** | 너무 길어서 중요한 규칙이 묻힘 → Claude가 절반 무시 | 가지치기. 이미 잘 하는 건 삭제하거나 hook으로 전환 |
| **검증 없는 신뢰** | 그럴듯하지만 엣지 케이스 미처리 | 항상 검증 수단 제공. 검증 못하면 배포하지 마라 |
| **무한 탐색** | 스코프 없는 "조사해봐" → 수백 파일 읽기 → 컨텍스트 폭발 | 범위를 좁히거나 서브에이전트에 위임 |

## 9. 직관 키우기

위 패턴은 출발점이지 절대 규칙이 아니다.

- 때로는 컨텍스트를 쌓아야 한다 — 하나의 복잡한 문제에 깊이 빠져 있을 때
- 때로는 계획을 건너뛰어야 한다 — 탐색적 작업일 때
- 때로는 모호한 프롬프트가 맞다 — Claude가 문제를 어떻게 해석하는지 보고 싶을 때

**좋은 결과가 나왔을 때 주목하라:** 프롬프트 구조, 제공한 컨텍스트, 사용한 모드.
**나쁜 결과가 나왔을 때 원인을 분석하라:** 노이즈? 모호함? 너무 큰 작업?

> [insight] "같은 이슈를 2번 이상 교정했으면 `/clear`하고 더 나은 프롬프트로 재시작하라"는 규칙이 핵심이다. 교정이 쌓이면 실패한 접근이 컨텍스트를 오염시키고 Claude가 과거 시도에 끌려간다. 하네스 설계 시 자동 재시도 로직을 만든다면, 실패 컨텍스트를 계속 누적하는 대신 클린 세션으로 전환하는 전략이 필요하다.

> [insight] CLAUDE.md의 준수율을 높이는 실전 기법: "IMPORTANT", "YOU MUST" 같은 강조 표현으로 준수율 향상 가능. 역으로, CLAUDE.md가 길어지면 중요한 규칙이 묻혀서 오히려 준수율이 하락한다. 규칙 추가보다 가지치기가 더 효과적인 경우가 많다.

> [insight] Hook과 CLAUDE.md의 결정적 차이: CLAUDE.md는 Claude가 읽고 따르려고 "노력"하는 가이드라인이고, Hook은 Claude 판단과 무관하게 **결정론적으로 실행**되는 스크립트다. "매번 예외 없이 실행되어야 하는 작업"은 Hook으로, "가이드라인"은 CLAUDE.md로 분리해야 한다.
