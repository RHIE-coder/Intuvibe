# Prompt Caching

---

반복 프롬프트 접두사를 캐시해 처리 비용과 레이턴시 대폭 감소. 정적 콘텐츠(시스템 프롬프트, 툴 정의, 대형 문서)에 특히 효과적.

- ZDR: ✅ 지원
- 전 활성 Claude 모델 지원

---

## 1. 활성화 방법

### 자동 캐싱 (권장)
```python
response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    cache_control={"type": "ephemeral"},  # 최상위 레벨
    system="...",
    messages=[...]
)
```
마지막 캐시 가능 블록에 자동으로 중단점 적용. 대화가 증가하면 앞으로 이동.

### 명시적 캐시 중단점
```python
system=[
    {"type": "text", "text": "정적 지시문", "cache_control": {"type": "ephemeral"}},
    {"type": "text", "text": "50페이지 문서...", "cache_control": {"type": "ephemeral"}}
]
```
최대 4개 중단점. 접두사 순서: `tools` → `system` → `messages`.

---

## 2. 비용

| 항목 | Opus 4.6 | Sonnet 4.6 | Haiku 4.5 |
|------|---------|-----------|---------|
| 기본 입력 | $5/MTok | $3/MTok | $1/MTok |
| 5분 캐시 쓰기 | $6.25 (+25%) | $3.75 | $1.25 |
| 1시간 캐시 쓰기 | $10 (+100%) | $6 | $2 |
| 캐시 읽기 | $0.50 (10%) | $0.30 | $0.10 |

**공식**: 캐시 읽기 = 기본 입력의 10%

---

## 3. TTL

- **기본**: 5분 (`{"type": "ephemeral"}`)
- **확장**: 1시간 (`{"type": "ephemeral", "ttl": "1h"}`)

**혼합 TTL 규칙**: 더 긴 TTL이 먼저 나열 (1h → 5m 순서)

---

## 4. 최소 캐시 가능 길이

| 모델 | 최소 토큰 |
|------|---------|
| Opus 4.6, Opus 4.5, Haiku 4.5 | 4,096 |
| Sonnet 4.6 | 2,048 |
| Sonnet 4.5, Opus 4.1, Opus 4 | 1,024 |
| Haiku 3.5 | 2,048 |

미달 시 캐시 없이 처리 (에러 없음, 조용히 미캐시).

---

## 5. 캐시 무효화 계층

| 변경 항목 | 영향 |
|---------|------|
| 툴 정의 수정 | 전체 캐시 무효화 |
| web_search 토글 | 시스템+메시지 캐시 무효화 |
| tool_choice 변경 | 메시지 캐시만 영향 |
| thinking 파라미터 | 메시지 캐시만 영향 |

**주의**: 키 순서 변경도 캐시 미스 유발 (Swift, Go 등 JSON 직렬화 시 키 순서 주의)

---

## 6. 사용량 추적

```python
print(f"캐시 쓰기: {response.usage.cache_creation_input_tokens}")
print(f"캐시 읽기: {response.usage.cache_read_input_tokens}")
print(f"일반 입력: {response.usage.input_tokens}")
# input_tokens는 마지막 캐시 중단점 이후 토큰만 포함
```

---

## 7. Lookback 윈도우

캐시 읽기 시 중단점당 최대 **20블록**을 뒤로 검색. 대화가 20블록 이상 증가하면 캐시 히트 보장을 위해 중단점이 필요.

---

## 8. 캐시 불가 항목

- thinking 블록 (직접 `cache_control` 불가; 이전 어시스턴트 턴으로는 캐시 가능)
- 빈 텍스트 블록
- 인용 등 서브 콘텐츠 블록

---

## 9. 워크스페이스 격리 (2026-02-05~)

조직 레벨 → 워크스페이스 레벨 격리로 변경. 동일 조직 내 워크스페이스 간 캐시 공유 불가. 정확한 일치(텍스트+이미지 100% 동일) 필요.

---

> [insight] 자동 캐싱(`cache_control` 최상위 설정)은 하네스의 멀티턴 대화에서 zero-effort 비용 최적화다. 명시적 중단점 없이도 대화가 증가할 때마다 캐시 중단점이 자동으로 앞으로 이동한다.

> [insight] 최소 캐시 토큰 요구사항(Opus: 4,096토큰)은 하네스에서 짧은 시스템 프롬프트의 캐싱 무효화를 조용히 실패하게 만든다. 캐시 효과 여부를 `cache_creation_input_tokens` + `cache_read_input_tokens`로 모니터링하고, 두 값이 모두 0이면 최소 길이 미달.

> [insight] 툴 정의 변경이 전체 캐시를 무효화한다는 점은 하네스의 플러그인 마켓플레이스에서 중요하다. 사용자가 플러그인을 추가/제거할 때마다 전체 캐시가 무효화되므로, 자주 사용하는 핵심 툴 세트는 변경하지 않고 나머지를 `defer_loading`으로 처리하는 전략이 필요하다.

> [insight] 1시간 TTL과 5분 TTL 혼합 패턴은 하네스의 다층 캐싱 전략에서 유용하다. 시스템 프롬프트처럼 거의 변하지 않는 콘텐츠는 1h, 대화 맥락은 5m으로 설정하면 비용과 신선도의 균형을 맞출 수 있다.

> [insight] 2026-02-05부터 워크스페이스 레벨 격리로 변경된 점은 하네스에서 멀티테넌트 아키텍처 설계 시 중요하다. 같은 조직 내 다른 워크스페이스(예: 개발/운영 환경)의 캐시가 공유되지 않으므로, 환경별 캐시 효율을 별도로 측정해야 한다.
