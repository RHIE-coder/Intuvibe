# Agent Skills Enterprise

---

엔터프라이즈 규모에서의 Skills 거버넌스, 보안 검토, 평가, 배포 관리.

---

## 1. 보안 리스크 평가

| 위험 지표 | 확인 항목 | 위험도 |
|---------|---------|------|
| 코드 실행 | `.py`, `.sh`, `.js` 스크립트 존재 | 높음 |
| 지시문 조작 | 안전 규칙 무시, 행동 숨김 지시 | 높음 |
| MCP 서버 참조 | `ServerName:tool_name` 형식 툴 참조 | 높음 |
| 네트워크 접근 | URL, `fetch`, `curl`, `requests` 호출 | 높음 |
| 하드코딩 자격증명 | API 키, 토큰, 비밀번호 | 높음 |
| 파일시스템 범위 | Skill 디렉토리 외부 경로, `../` 경로 순회 | 중간 |

**검토 체크리스트**: 전체 파일 읽기 → 스크립트 샌드박스 실행 → 적대적 지시문 확인 → 외부 URL 검증 → 자격증명 검색 → 데이터 유출 패턴 확인

---

## 2. 평가 게이트

배포 전 필수 평가 차원:

| 차원 | 측정 항목 | 실패 예시 |
|------|---------|---------|
| 트리거 정확성 | 올바른 쿼리에만 활성화 | 무관한 질문에도 트리거 |
| 격리 동작 | 단독 작동 정상성 | 존재하지 않는 파일 참조 |
| 공존성 | 다른 Skills 성능 영향 없음 | 광범위한 description으로 트리거 탈취 |
| 지시문 준수 | 절차 정확히 수행 | 검증 단계 건너뜀 |
| 출력 품질 | 정확하고 유용한 결과 | 포맷 오류, 누락 데이터 |

**요구사항**: Skill당 3-5개 쿼리 (트리거 O, 트리거 X, 엣지 케이스 포함). 사용 모델(Haiku/Sonnet/Opus) 전체 테스트 필수.

---

## 3. Skill 생명주기 (6단계)

**계획 → 생성+검토 → 테스트 → 배포 → 모니터링 → 반복/폐기**

- **검토**: Skill 작성자 ≠ 검토자 (역할 분리)
- **배포**: Skills API로 워크스페이스 전체 배포
- **모니터링**: API에 사용 분석 없음 → 애플리케이션 레벨 로깅 직접 구현
- **폐기**: 평가 지속 실패 또는 워크플로우 폐기 시

---

## 4. 규모 관리

### 동시 로딩 한도
- 요청당 최대 **8개 Skills** (API 제한)
- Skills 수 증가 시 리콜 정확도 측정 → 성능 저하 전에 중단
- 8개 초과 필요 시: 관련 Skills 통합 또는 작업 유형별 라우팅

### 통합 전략
- 시작: 좁은 워크플로우별 Skills (`formatting-sales-reports`)
- 통합: 역할 기반 번들 (`sales-operations`) — 평가로 동등 성능 확인 후

### 역할 기반 번들 예시
- 영업팀: CRM 운영, 파이프라인 리포팅
- 개발팀: 코드 리뷰, 배포 워크플로우
- 재무팀: 리포트 생성, 감사 준비

---

## 5. 배포 및 버전 관리

- **소스 컨트롤**: Skill 디렉토리를 Git으로 관리 (PR 기반 검토, 롤백)
- **프로덕션**: 특정 버전 고정, 신규 버전은 전체 평가 후 승격
- **롤백**: 이전 버전 항상 유지, 실패 즉시 롤백
- **무결성**: 검토된 Skills의 체크섬 계산 → 배포 시 검증, 서명 커밋 사용
- **크로스 서페이스**: 플랫폼 간 자동 동기화 없음 → Git을 단일 진실 소스로, 별도 동기화 프로세스 구현

---

> [insight] 요청당 최대 8개 Skills 제한은 하네스의 플러그인 마켓플레이스에서 중요한 설계 제약이다. 사용자가 많은 플러그인을 활성화하려면 task-type 기반 라우팅(코딩 요청 → 개발 Skills 세트, 문서 요청 → 문서 Skills 세트)이 필수다. 단순히 모든 Skills를 활성화하는 방식은 API 한도와 리콜 정확도 양쪽에서 문제가 된다.

> [insight] "API에 사용 분석 없음" 제약은 하네스에서 직접 애플리케이션 레벨 로깅을 구현해야 한다는 의미다. 어떤 Skills가 어떤 요청에 트리거됐는지, 성공/실패 비율, 모델별 성능 차이를 하네스에서 직접 추적해야 한다. 이는 플러그인 마켓플레이스의 품질 관리와 과금 모델에도 기반 데이터가 된다.

> [insight] 역할 분리(Skill 작성자 ≠ 검토자) 원칙은 하네스의 플러그인 마켓플레이스 거버넌스에서 핵심이다. 서드파티 플러그인 생태계를 운영한다면, 플러그인 제출 → 보안 감사(별도 팀) → 평가 테스트 → 승인/거부 파이프라인이 마켓플레이스 신뢰성의 기반이 된다.
