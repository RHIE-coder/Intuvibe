# Reducing Latency

---

**원칙**: 먼저 성능이 좋은 프롬프트를 만들고, 그 다음에 레이턴시 최적화. 조기 최적화는 최대 성능을 발견하는 것을 방해함.

---

## 1. 레이턴시 측정 지표

| 지표 | 설명 |
|------|------|
| **Baseline Latency** | 프롬프트 처리 + 전체 응답 생성까지 총 시간 |
| **TTFT (Time to First Token)** | 프롬프트 전송 후 첫 번째 토큰 수신까지 시간. 스트리밍 UX에서 핵심 지표 |

---

## 2. 레이턴시 감소 기법

### ① 적절한 모델 선택

속도 우선 → **Claude Haiku 4.5** (가장 빠른 응답 + 높은 지능)

```python
client.messages.create(
    model="claude-haiku-4-5",
    max_tokens=100,
    messages=[{"role": "user", "content": "..."}],
)
```

### ② 프롬프트 & 출력 길이 최적화

| 기법 | 설명 | 주의사항 |
|------|------|--------|
| 간결한 지시 | 불필요한 세부사항 제거 | 컨텍스트 부족으로 논리 도약 실패 주의 |
| 짧은 응답 요청 | Claude에게 직접 간결하게 답하라고 지시 | 단어 수 제한보다 문단/문장 수 제한이 더 효과적 |
| `max_tokens` 설정 | 출력 최대 길이 하드 제한 | 문장 중간에 잘릴 수 있음 → 후처리 필요 |
| `temperature` 조정 | 낮은 값(0.2)→ 더 집중되고 짧은 응답 경향 | 높은 값(0.8)→ 다양하지만 길어질 수 있음 |

### ③ 스트리밍 활용

모델이 전체 응답 완성 전 토큰을 즉시 전송 → 사용자가 실시간으로 응답 확인.  
TTFT 개선으로 체감 반응성 크게 향상.

---

> [insight] 하네스의 플러그인 실행 환경에서 레이턴시 프로파일은 사용 사례에 따라 전략이 달라진다. 사용자 대면 실시간 플러그인은 Haiku + 스트리밍 + 짧은 `max_tokens` 조합을 기본으로 제공하고, 백그라운드 배치 처리 플러그인은 Sonnet/Opus + 높은 `max_tokens`를 허용하는 방식으로 플러그인 설정 스키마에 `latency_profile: "realtime" | "batch"` 옵션을 추가하는 것이 타당하다.
