# Manage Tool Context

---

툴 정의와 `tool_result` 누적으로 인한 컨텍스트 윈도우 소진을 방지하는 4가지 접근법 개요.

---

## 4가지 접근법

| 접근법 | 줄이는 대상 | 적합한 상황 |
|--------|-----------|-----------|
| Tool search | 업프론트 툴 정의 토큰 | 툴 20개 이상, 매 턴 대부분 불필요 |
| Programmatic tool calling | `tool_result` 라운드트립 | 연속 툴 호출을 단일 스크립트로 처리 가능할 때 |
| Prompt caching | 반복 툴 정의 비용 | 안정적인 툴셋이 대량 요청에 재사용될 때 |
| Context editing | 대화 히스토리의 오래된 `tool_result` | 긴 대화에서 초반 결과가 더 이상 불필요할 때 |

---

## 각 접근법 요약

**Tool search**: 툴 50개 스키마를 처음부터 전달하지 않고, `tool_search` 툴 하나만 전달. Claude가 필요 시 온디맨드로 검색. 레이턴시 소량 증가 대신 기본 컨텍스트 사용량 대폭 감소.

**Programmatic tool calling**: 5번의 `tool_use`↔`tool_result` 라운드트립 대신, Claude가 코드 블록 하나를 작성하고 Anthropic 샌드박스가 실행. 중간 결과가 대화 히스토리에 진입하지 않음.

**Prompt caching**: 토큰 수를 줄이지는 않지만 반복 비용을 줄임. 툴 정의를 한 번 캐시하면 수천 요청에서 재사용. 큰 고정 툴셋에 적합.

**Context editing**: 더 이상 필요 없는 과거 `tool_result` 블록을 대화 히스토리에서 제거. 에이전트 루프 수백 턴 후 초반 결과 정리.

---

## 단계별 적용 전략

장기 고용량 에이전트를 위한 권장 도입 순서:

1. **즉시**: 프롬프트 캐싱 활성화 (캐시 쓰기 25% 추가 비용, 두 번째 요청부터 회수)
2. **툴 20개 초과 시**: Tool search 추가
3. **대화가 길어질 때**: Context editing 추가
4. **반복적 소형 툴 호출 체인 발견 시**: Programmatic tool calling 고려

4가지 접근법은 서로 다른 문제를 해결하므로 조합 사용 가능. 충돌 없음.

---

> [insight] 이 페이지는 하네스의 컨텍스트 관리 아키텍처 결정 트리를 명확히 제시한다. 툴셋 크기·안정성·대화 길이·호출 패턴에 따라 4가지 접근법 중 적합한 것을 선택하는 로직을 하네스의 구성 레이어에서 제공할 수 있다.

> [insight] "툴 20개" 기준점은 하네스의 플러그인 마켓플레이스 설계에서 중요한 임계값이다. 플러그인이 20개를 초과하면 Tool search 없이는 기본 컨텍스트 사용량이 급증하므로, 마켓플레이스에서 활성화된 플러그인 수를 모니터링하고 임계값 초과 시 자동으로 Tool search 모드로 전환하는 로직이 필요하다.

> [insight] 프롬프트 캐싱의 "즉시 적용" 권장은 하네스 설계에서 default-on 전략을 의미한다. 툴 정의가 요청마다 동일하다면 (대부분의 경우) 캐싱을 기본으로 켜두고, 캐시 쓰기 25% 추가 비용이 두 번째 요청에서 이미 회수된다는 점을 비용 모델에 포함해야 한다.
