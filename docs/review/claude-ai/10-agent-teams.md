# Agent Teams — 다중 세션 협업 오케스트레이션

---

**실험적 기능 (기본 비활성화)**. 여러 Claude Code 인스턴스가 팀으로 협업한다.
하나의 세션이 Team Lead, 나머지가 Teammate로 동작하며 공유 작업 목록 + 직접 메시징으로 조율한다.

활성화:
```json
// settings.json
{ "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" } }
```

---

## 1. Subagent vs Agent Team

| | Subagent | Agent Team |
|---|---|---|
| **컨텍스트** | 자체 윈도우, 결과를 호출자에게 반환 | 자체 윈도우, 완전 독립 |
| **통신** | 메인에만 보고 | 팀원 간 직접 메시지 |
| **조율** | 메인이 관리 | 공유 작업 목록으로 자기 조율 |
| **적합한 상황** | 결과만 중요한 집중 작업 | 토론, 발견 공유, 상호 도전이 필요한 복잡한 작업 |
| **토큰 비용** | 낮음 (요약만 반환) | 높음 (각 팀원이 별도 인스턴스) |

---

## 2. 적합한 상황

- **리서치/리뷰** — 여러 관점에서 동시 탐색, 발견 공유/도전
- **신규 모듈/기능** — 각 팀원이 별도 파일 담당
- **경쟁 가설 디버깅** — 각자 다른 이론을 병렬 검증, 상호 반박
- **크로스 레이어** — 프론트엔드/백엔드/테스트를 각각 담당

**부적합:** 순차 작업, 같은 파일 편집, 의존성 많은 작업 → 단일 세션이나 subagent가 효율적.

---

## 3. 아키텍처

| 컴포넌트 | 역할 |
|---------|------|
| **Team Lead** | 팀 생성, 팀원 스폰, 작업 조율을 담당하는 메인 세션 |
| **Teammates** | 각각 독립 Claude Code 인스턴스 |
| **Task list** | 공유 작업 목록. pending → in progress → completed. 의존성 자동 관리 |
| **Mailbox** | 에이전트 간 메시징 시스템 |

저장 위치:
- 팀 설정: `~/.claude/teams/{team-name}/config.json` (런타임 상태, 수동 편집 불가)
- 작업 목록: `~/.claude/tasks/{team-name}/`

> 프로젝트 레벨 팀 설정(`.claude/teams/`)은 인식되지 않음. 재사용 역할은 subagent 정의로.

---

## 4. 디스플레이 모드

| 모드 | 설명 | 요구사항 |
|------|------|---------|
| **in-process** | 메인 터미널에서 전부 실행. `Shift+Down`으로 팀원 순환 | 없음 |
| **split panes** | 각 팀원이 별도 pane. 동시 출력 확인 가능 | tmux 또는 iTerm2 |
| **auto** (기본) | tmux 세션 안이면 split, 아니면 in-process | - |

```json
// ~/.claude.json
{ "teammateMode": "in-process" }
```

또는 `claude --teammate-mode in-process`

## 5. 팀 생성과 조율

### 자연어로 팀 생성

```
CLI 도구 설계를 다각도로 탐색할 에이전트 팀을 만들어.
UX 1명, 기술 아키텍처 1명, devil's advocate 1명.
```

### 팀원 수/모델 지정

```
4명의 팀원으로 모듈 병렬 리팩토링. 각 팀원은 Sonnet 사용.
```

### 계획 승인 요구

```
인증 모듈 리팩토링할 아키텍트 팀원을 스폰해.
변경 전에 계획 승인을 받도록 해.
```

팀원이 plan mode에서 계획 완성 → lead에게 승인 요청 → 승인 시 구현 시작, 거부 시 피드백 반영 후 재제출.
Lead의 판단 기준은 프롬프트로 제어: "테스트 커버리지가 포함된 계획만 승인해"

### 팀원과 직접 대화

- **in-process**: `Shift+Down`으로 순환 → 입력으로 메시지. `Enter`로 세션 보기, `Esc`로 중단, `Ctrl+T`로 작업 목록 토글
- **split panes**: 해당 pane 클릭

### 작업 할당/자기 클레임

- Lead가 명시 할당 또는 팀원이 미할당/미차단 작업을 자기 클레임
- 파일 잠금으로 동시 클레임 경합 방지
- 작업 상태: pending → in progress → completed
- 의존성: 선행 작업 완료 전까지 차단

### 종료

```
researcher 팀원에게 종료 요청해
```

팀원이 승인/거부 가능. 전체 정리:

```
팀 정리해
```

활성 팀원이 있으면 실패 → 먼저 종료. **항상 lead가 정리** (팀원이 하면 불일치 위험).

## 6. Subagent 정의를 팀원으로 사용

```
security-reviewer 에이전트 타입으로 팀원 스폰해서 인증 모듈 감사해.
```

- subagent의 `tools`와 `model` 적용
- 본문은 팀원 시스템 프롬프트에 **추가** (대체가 아님)
- SendMessage 등 팀 조율 도구는 tools 제한과 무관하게 항상 사용 가능
- **주의:** `skills`와 `mcpServers` frontmatter는 팀원으로 실행 시 적용되지 않음 (프로젝트/사용자 설정에서 로드)

---

## 7. 컨텍스트와 통신

- 팀원은 lead의 대화 히스토리를 상속하지 않음
- CLAUDE.md, MCP, Skills는 일반 세션처럼 로드
- 스폰 프롬프트가 초기 컨텍스트

**통신 수단:**
- **message** — 특정 팀원에게 1:1 전송
- **broadcast** — 전체 팀원에게 전송 (비용이 팀 크기에 비례, 아껴 사용)
- 자동 메시지 전달: 팀원 → lead 자동 도착
- idle 알림: 팀원 완료 시 lead에 자동 통보
- 공유 작업 목록으로 상태 확인/클레임

Lead가 팀원에 이름을 부여. 팀원 간 이름으로 메시지 가능.

---

## 8. Hooks

| 훅 | 발화 시점 | exit 2 동작 |
|----|---------|------------|
| `TeammateIdle` | 팀원이 idle 상태로 전환 직전 | 피드백 전달 + 팀원 계속 작업 |
| `TaskCreated` | 작업 생성 시 | 생성 방지 + 피드백 |
| `TaskCompleted` | 작업 완료 마킹 시 | 완료 방지 + 피드백 |

---

## 9. 권한

- 팀원은 lead의 권한 설정으로 시작
- `--dangerously-skip-permissions`면 전체 팀원도 동일
- 스폰 후 개별 변경 가능, 스폰 시점에는 불가

---

## 10. 베스트 프랙티스

| 원칙 | 상세 |
|------|------|
| **충분한 컨텍스트 제공** | 팀원은 lead 대화 히스토리 없음. 스폰 프롬프트에 구체적 정보 포함 |
| **팀 규모 3-5명** | 토큰 선형 증가 + 조율 오버헤드. 5-6 tasks/teammate가 적정 |
| **적절한 작업 크기** | 너무 작으면 오버헤드 > 이점, 너무 크면 체크인 없이 오래 작업 → 낭비 위험 |
| **팀원 완료 대기** | Lead가 직접 구현하려 하면 "팀원 완료를 기다려"로 지시 |
| **리서치/리뷰부터 시작** | 코드 작성 없는 작업(PR 리뷰, 라이브러리 조사)이 입문에 적합 |
| **파일 충돌 회피** | 같은 파일을 두 팀원이 편집 → 덮어쓰기. 담당 파일 분리 |
| **주기적 모니터링** | 무인 장시간 실행은 낭비 위험. 진행 상황 확인 + 방향 수정 |

## 11. 사용 예시

### 병렬 코드 리뷰

```
PR #142를 리뷰할 에이전트 팀 생성.
- 보안 영향 집중 1명
- 성능 영향 확인 1명
- 테스트 커버리지 검증 1명
각각 리뷰 후 결과 보고.
```

단일 리뷰어는 한 유형에 치우침 → 독립 도메인 분리로 균형 잡힌 리뷰.

### 경쟁 가설 디버깅

```
앱이 첫 메시지 후 종료되는 문제.
5명의 팀원으로 서로 다른 가설 조사.
서로 대화하면서 상대 이론을 반박하려 시도해, 과학적 토론처럼.
합의된 결론을 문서에 업데이트.
```

순차 조사는 앵커링 편향 → 병렬 독립 조사 + 상호 반박 = 실제 원인 생존 확률 높음.

---

## 12. 알려진 제한사항

- `/resume`으로 in-process 팀원 복원 불가
- 작업 상태가 지연될 수 있음 (팀원이 완료 마킹 누락)
- 종료가 느릴 수 있음 (현재 도구 호출 완료까지 대기)
- 세션당 1팀만
- 중첩 팀 불가 (팀원이 팀 생성 불가)
- Lead 고정 (승격/이전 불가)
- 스폰 시 개별 권한 모드 설정 불가
- split panes: VS Code 통합 터미널, Windows Terminal, Ghostty 미지원

> [insight] Agent Teams의 핵심 차별점은 **팀원 간 직접 메시지**다. Subagent는 메인에만 보고하지만 agent team 팀원은 서로 대화하면서 발견을 공유하고 상대 이론을 반박할 수 있다. "경쟁 가설 디버깅" 패턴이 이를 가장 잘 활용하는 사례다. 순차 조사의 앵커링 편향을 병렬 + 상호 반박으로 극복.

> [insight] 팀 설정(`~/.claude/teams/`)은 런타임 상태 파일이라 수동 편집/사전 작성이 의미 없다 (다음 상태 업데이트에 덮어씀). 프로젝트 레벨(`.claude/teams/`)도 인식 안 됨. 재사용 가능한 팀원 역할은 subagent 정의(`.claude/agents/`)로 만들고 팀 생성 시 참조하는 것이 정석.

> [insight] 팀원이 subagent 정의를 사용할 때 `skills`와 `mcpServers` frontmatter가 무시된다. 이는 subagent 단독 실행과 팀원으로 실행할 때 동작이 달라진다는 의미. 하네스에서 같은 에이전트 정의를 양쪽에서 쓴다면 이 차이를 인지해야 한다.
