# How Claude Code Works

---

## 개요

Claude Code는 터미널에서 실행되는 **에이전틱 어시스턴트**다. 코딩뿐 아니라 커맨드라인에서 할 수 있는 모든 것(문서 작성, 빌드 실행, 파일 검색, 리서치 등)을 수행한다.

---

## 1. Agentic Loop (에이전틱 루프)

Claude에게 작업을 주면 3단계를 거친다:

```
[Your Prompt] → Gather Context → Take Action → Verify Results → (반복) → 완료
                      ↑                                              |
                      └──────── 언제든 사용자가 개입 가능 ───────────┘
```

- 3단계는 명확히 분리되지 않고 **블렌딩**됨
- 질문 → context gathering만 필요
- 버그 수정 → 3단계 전부 반복
- 리팩토링 → 검증 단계가 길어짐
- 사용자는 **언제든 interrupt** 가능 (입력 후 Enter)

에이전틱 루프는 두 컴포넌트로 구동:
- **Models** — 추론 담당
- **Tools** — 행동 담당

> Claude Code는 Claude 모델을 감싸는 **agentic harness** 역할을 한다. 도구, 컨텍스트 관리, 실행 환경을 제공하여 LLM을 코딩 에이전트로 만든다.

### 1-1. Models

- 코드를 읽고, 컴포넌트 연결 관계를 이해하고, 목표 달성을 위해 무엇을 변경할지 판단
- 복잡한 작업은 단계별로 분해 → 실행 → 학습 기반 조정
- **Sonnet** — 대부분의 코딩 작업에 적합
- **Opus** — 복잡한 아키텍처 결정에 강한 추론력
- 모델 전환: 세션 중 `/model` 또는 `claude --model <name>`

### 1-2. Tools (도구)

도구가 없으면 Claude는 텍스트 응답만 가능. 도구가 있어야 **에이전틱**해진다.

| 카테고리 | 할 수 있는 것 |
|---------|-------------|
| **File operations** | 파일 읽기, 코드 편집, 새 파일 생성, 이름 변경 및 재구성 |
| **Search** | 패턴으로 파일 찾기, 정규식으로 내용 검색, 코드베이스 탐색 |
| **Execution** | 셸 명령 실행, 서버 시작, 테스트 실행, git 사용 |
| **Web** | 웹 검색, 문서 가져오기, 에러 메시지 조회 |
| **Code intelligence** | 편집 후 타입 에러/경고 확인, 정의로 이동, 참조 찾기 (플러그인 필요) |

예시 — "실패하는 테스트 수정해줘":
1. 테스트 실행 → 실패 확인
2. 에러 출력 읽기
3. 관련 소스 파일 검색
4. 파일 읽고 코드 이해
5. 파일 편집해서 수정
6. 테스트 재실행으로 검증

**확장 수단:**
- [Skills](/en/skills) — 워크플로우 정의
- [MCP](/en/mcp) — 외부 서비스 연결
- [Hooks](/en/hooks) — 워크플로우 자동화
- [Sub-agents](/en/sub-agents) — 작업 위임

## 2. Claude가 접근할 수 있는 것

`claude`를 디렉토리에서 실행하면 다음에 접근 가능:

| 대상 | 설명 |
|------|------|
| **프로젝트** | 현재 디렉토리 및 하위 파일 (허가 시 외부 파일도) |
| **터미널** | 빌드 도구, git, 패키지 매니저, 시스템 유틸리티, 스크립트 등 |
| **Git 상태** | 현재 브랜치, 커밋되지 않은 변경, 최근 커밋 히스토리 |
| **CLAUDE.md** | 프로젝트별 지침, 규칙, 컨텍스트 |
| **Auto memory** | 작업하면서 자동 저장된 학습 (패턴, 선호도). 세션 시작 시 MEMORY.md의 처음 200줄 또는 25KB 로드 |
| **확장** | MCP 서버, Skills, Sub-agents, Chrome 통합 |

> 인라인 코드 어시스턴트(현재 파일만 보는)와 달리 **프로젝트 전체**를 보고 작업할 수 있다.

## 3. 실행 환경 & 인터페이스

### 실행 환경

| 환경 | 코드 실행 위치 | 용도 |
|------|-------------|------|
| **Local** | 내 머신 | 기본값. 파일, 도구, 환경에 대한 전체 접근 |
| **Cloud** | Anthropic 관리 VM | 작업 오프로드, 로컬에 없는 레포 작업 |
| **Remote Control** | 내 머신 (브라우저에서 제어) | 웹 UI 사용하면서 로컬 유지 |

### 인터페이스

터미널, Desktop App, IDE 확장(VS Code, JetBrains), claude.ai/code, Remote Control, Slack, CI/CD 파이프라인 — **에이전틱 루프는 동일**.

## 4. 세션 관리

### 기본 개념
- 대화가 **로컬에 자동 저장**됨 (메시지, 도구 사용, 결과 모두)
- 코드 변경 전 **파일 스냅샷** 생성 → 되돌리기 가능
- **세션은 독립적** — 새 세션은 이전 대화 히스토리 없이 시작

### 브랜치 작업
- 세션은 **현재 디렉토리에 묶임**
- 브랜치 전환 시 새 브랜치 파일을 보지만, 대화 히스토리는 유지
- 병렬 작업: **git worktree** 사용 → 별도 디렉토리에서 독립 세션

### Resume vs Fork

| 명령 | 동작 |
|------|------|
| `claude --continue` (`-c`) | 현재 디렉토리의 가장 최근 세션 이어서 |
| `claude --resume` (`-r`) | 특정 세션 선택해서 재개 |
| `claude --continue --fork-session` | 대화 히스토리 복사 + 새 세션 ID 생성 (원본 유지) |

> 같은 세션을 여러 터미널에서 resume하면 메시지가 뒤섞인다. 병렬 작업은 `--fork-session` 사용.

## 5. Context Window (컨텍스트 윈도우)

컨텍스트에 담기는 것:
- 대화 히스토리, 파일 내용, 명령 출력
- CLAUDE.md, Auto memory, 로드된 Skills, 시스템 지침

### 컨텍스트가 가득 찰 때
1. 오래된 도구 출력을 먼저 제거
2. 필요 시 대화 요약
3. 요청과 핵심 코드 스니펫은 보존
4. 초기 대화의 상세 지침은 손실될 수 있음

**대응 방법:**
- 지속적인 규칙은 **CLAUDE.md**에 기록
- `/compact [focus]` — 포커스 지정 압축 (예: `/compact focus on the API changes`)
- `/context` — 현재 공간 사용량 확인
- `/mcp` — 서버별 비용 확인

### Skills & Sub-agents로 컨텍스트 관리
- **Skills** — 온디맨드 로드. `disable-model-invocation: true`로 수동 호출 전까지 컨텍스트에서 제외
- **Sub-agents** — 별도의 독립 컨텍스트. 메인 대화를 부풀리지 않음. 완료 시 요약만 반환

## 6. 안전장치

### Checkpoints (체크포인트)
- 파일 편집 전 **스냅샷 자동 생성**
- `Esc` 두 번 → 이전 상태로 되돌리기
- 로컬 세션 한정, git과 별개
- **파일 변경만 커버** — 원격 시스템(DB, API, 배포)에 대한 액션은 체크포인트 불가

### Permissions (권한)
`Shift+Tab`으로 모드 전환:

| 모드 | 동작 |
|------|------|
| **Default** | 파일 편집, 셸 명령 모두 승인 필요 |
| **Auto-accept edits** | 파일 편집은 자동, 명령은 승인 필요 |
| **Plan mode** | 읽기 전용 도구만 사용, 실행 전 계획 승인 |
| **Auto mode** | 모든 액션을 백그라운드 안전 검사와 함께 평가 (리서치 프리뷰) |

신뢰하는 명령은 `.claude/settings.json`에서 허용 목록에 추가 가능.

## 7. 효과적으로 사용하는 팁

### Claude에게 사용법 물어보기
- "hooks 어떻게 설정해?" 같은 질문 가능
- `/init` — CLAUDE.md 생성 가이드
- `/agents` — 커스텀 서브에이전트 설정
- `/doctor` — 설치 문제 진단

### 대화형으로 작업
- 완벽한 프롬프트 불필요. 시작 후 반복 수정
- 잘못된 방향이면 그냥 수정 사항 입력 → Claude가 조정

### 구체적으로 요청
```
체크아웃 플로우가 만료된 카드 사용자에게 깨져 있어.
src/payments/ 확인해봐, 특히 토큰 리프레시.
실패하는 테스트 먼저 작성하고, 그 다음 수정해.
```
> 모호한 프롬프트도 작동하지만, 구체적일수록 첫 시도에 성공할 확률이 높다.

### 검증 수단 제공
```
validateEmail 구현해. 테스트 케이스:
'user@example.com' → true, 'invalid' → false, 'user@.com' → false.
테스트도 실행해.
```

### 탐색 후 구현
- 복잡한 문제는 **Plan mode**(`Shift+Tab` 2번)로 먼저 분석
- 계획 리뷰 → 대화로 다듬기 → 구현

### 위임하되, 지시하지 말 것
```
체크아웃 플로우가 만료된 카드에서 깨져 있어.
관련 코드는 src/payments/에 있어. 조사해서 고쳐줘.
```
> 어떤 파일을 읽을지, 어떤 명령을 실행할지 지정할 필요 없다. Claude가 알아서 판단.
